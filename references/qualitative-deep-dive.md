# Qualitative Deep Dive

Use this for interview-heavy, open-text-heavy, or otherwise meaning-rich qualitative work. This is the main engine that prevents shallow theme summarization.

## Purpose

Do not jump from transcripts to polished findings. First build the intermediate assets that make the analysis credible, explainable, and vivid.

Use two tracks:
- Track A: theory-informed
- Track B: data-first

Default to Track B unless the user already has a strong theory frame or explicitly wants theory-informed analysis.

## Required phases

### Phase 0: Data cleaning, deduplication, and linkage

Before interpretation:
- identify duplicate respondents or repeated submissions
- flag low-information or invalid responses
- if survey data exists, confirm whether records can be linked
- preserve duplicates as analyzable cases when consistency itself is meaningful

Output:
- cleaning log
- linkage log
- flagged-record list

### Phase 1: Immersion

Read for story before collapsing into labels.

When volume is high:
- scan the full corpus for language patterns and sample distribution
- then sample deeply across meaningful groups and anomalies

Use sensitizing questions:
- What is the user trying to get done?
- What happened that made the moment hard, risky, confusing, or satisfying?
- What trade-off is the user making?
- Where do account and behavior seem misaligned?
- What changed over time?
- What context shapes the decision?

Output:
- immersion memos
- early observations
- repeated phrases and metaphors
- missing voices or blind spots

### Phase 2: Iterative coding

Minimum standard:
- Pass 1: open coding
- Pass 2: focused coding

Optional:
- Pass 3: theory coding, only if it clarifies rather than forces the data

Coding rules:
- keep code names concrete
- preserve source references
- track code merges, splits, and renames
- distinguish record frequency from unique-person frequency when possible

Output:
- codebook with definitions, boundaries, examples, and evolution notes
- coded excerpt list or coded case summaries

### Phase 3: Case comparison and patterning

Move from code clusters to meaningful user patterns.

Look for:
- repeated need states
- repeated workarounds
- repeated barriers
- repeated decision strategies
- notable within-group variation

Do not group users by demographics alone unless demographics clearly structure the behavior.

Output:
- pattern cards
- participant profiles or casebook
- typical, extreme, and deviant cases

### Phase 4: Contradictions, negative cases, and credibility

Treat contradictions as evidence, not clutter.

Check:
- contradictions within a single interview
- contradictions across repeated interviews from the same respondent
- disagreement between linked survey and interview data
- cases that weaken the main pattern

Output:
- contradiction notes
- negative-case notes
- respondent credibility flags when needed

### Phase 5: Synthesis for report writing

Only now draft findings.

For each major finding, preserve:
- the pattern
- the mechanism or interpretation
- the boundary condition
- supporting quotes or scenes
- counter-signals

## Mandatory assets

For substantial qualitative work, maintain:
- cleaning log
- immersion memo set
- codebook
- quote bank
- casebook or participant profiles
- contradiction / negative-case notes

If these do not exist, the analysis is probably not yet deep enough for a strong final report.

## Casebook guidance

Include at least three kinds of cases:
- representative cases
- extreme cases
- deviant or counter cases

Each case should capture:
- who the person is in decision-relevant terms
- what situation they were in
- what they tried to do
- what happened
- why the case matters analytically

## Quote bank guidance

Do not collect only catchy lines. Prefer quotes that reveal:
- decision logic
- uncertainty
- emotional turning points
- trade-offs
- misunderstandings
- workarounds

## Failure modes

- theme summaries with too few underlying cases
- relying on one vivid quote to stand in for a pattern
- skipping immersion and going straight to coding
- skipping coding and going straight to executive prose
- flattening contradictions into a neat but false story
